1. Initial Idea / Summary

When a promise is resolved with a normal value, it fulfills with that value. When it is resolved with another promise or a thenable object, JavaScript does not simply wrap that object and move on. It inspects it, follows it, and adopts its final state. This adoption behavior is called promise assimilation.

Core rule: A promise resolved with another promise or thenable tries to become whatever that inner async value eventually becomes.
TermMeaningTypical example
Resolution procedureThe internal algorithm that decides how a returned or resolved value affects a promise.resolve(x) or return x inside .then()
AssimilationAdopting the state of a promise-like value, especially another promise or thenable.Promise.resolve(fetch(...))
FlatteningThe observable result: promise chains behave like one level, not deeply nested boxes.Promise.resolve(Promise.resolve(42)) fulfills with 42

2. Mental Model

Think of a promise as a sealed container whose contents are not decided immediately. Resolving the container does not always mean fulfilling it. It means: “start the process of deciding this promise’s final fate.” Naturally JavaScript needed this to be subtle, because apparently one kind of async box was not enough for civilization.

Normal value

If the resolved value is not promise-like, the promise fulfills with that value.

Promise.resolve(10)
  .then(value => console.log(value));

// 10

Promise or thenable

If the resolved value has a callable then, JavaScript follows it and adopts what it settles to.

Promise.resolve(Promise.resolve(10))
  .then(value => console.log(value));

// 10

The three-question model

QuestionWhat happens
Is the value the same promise?Reject with a TypeError. A promise cannot resolve to itself.
Is the value an object or function with callable then?Call that then and adopt the result. This is thenable assimilation.
Otherwise?Fulfill with the value directly.

3. Detailed Examples

3.1 Simple value: no assimilation needed

Promise.resolve("Ada")
  .then(name => {
    console.log(name);
    return name.length;
  })
  .then(length => {
    console.log(length);
  });
Output:
Ada
3

The first promise fulfills with the string "Ada". The first .then() returns the plain number 3, so the next promise fulfills with 3. No promise-like object appears, so no adoption is needed.

3.2 Returning a promise from .then()

Promise.resolve("start")
  .then(value => {
    console.log(value);
    return Promise.resolve("inner value");
  })
  .then(value => {
    console.log(value);
  });
Output:
start
inner value

The first .then() does not pass the inner promise object as the next value. It returns a promise, so the promise created by .then() adopts that returned promise’s state. The next handler receives "inner value", not Promise { "inner value" }.

3.3 Promise flattening: nested promises do not behave like nested arrays

const nested = Promise.resolve(
  Promise.resolve(
    Promise.resolve(42)
  )
);

nested.then(value => {
  console.log(value);
});
Output:
42

Even though the source code looks nested, the observable result is flattened. JavaScript keeps adopting promise states until it reaches a non-promise fulfillment value or a rejection reason.

3.4 Thenable assimilation: promise-like objects count too

const thenable = {
  then(resolve) {
    resolve("value from a thenable");
  }
};

Promise.resolve(thenable)
  .then(value => console.log(value));
Output:
value from a thenable

The object is not created by new Promise(), but it has a callable then method. JavaScript treats it as thenable and assimilates it. This lets promises interoperate with promise-like libraries, which is useful and also a fine source of haunted edge cases.

3.5 Complex example: return, throw, catch, and assimilation

Promise.resolve("A")
  .then(value => {
    console.log(value);
    return Promise.resolve("B");
  })
  .then(value => {
    console.log(value);
    throw new Error("C failed");
  })
  .catch(error => {
    console.log(error.message);
    return Promise.resolve("D recovered");
  })
  .then(value => {
    console.log(value);
  });
Output:
A
B
C failed
D recovered

The first handler returns a fulfilled promise, so the chain waits for and adopts "B". The second handler throws, so the promise returned by that .then() rejects. The .catch() handles the rejection and returns another fulfilled promise. That returned promise is assimilated, so the final .then() receives "D recovered".

3.6 Order of execution with microtasks

console.log("sync 1");

Promise.resolve()
  .then(() => {
    console.log("then 1");
    return Promise.resolve("nested");
  })
  .then(value => {
    console.log("then 2", value);
  });

console.log("sync 2");
Output:
sync 1
sync 2
then 1
then 2 nested

Synchronous code runs first. Promise reaction callbacks run later as microtasks. Returning Promise.resolve("nested") causes the next promise in the chain to assimilate the returned promise before running the next handler.

3.7 Async functions use the same idea

async function getValue() {
  return Promise.resolve("from async");
}

getValue().then(value => {
  console.log(value);
});
Output:
from async

An async function always returns a promise. If you return another promise from it, the outer async function’s promise adopts the returned promise’s final state. So no, you do not get Promise<Promise<string>> in normal JavaScript behavior.

4. Gotchas and Tricky Associated Concepts

Gotcha 1: “Resolved” does not always mean “fulfilled.”
A promise can be resolved to another pending promise. In that case, it is locked into following the other promise but may still be pending.
let finish;

const inner = new Promise(resolve => {
  finish = resolve;
});

const outer = Promise.resolve(inner);

outer.then(value => console.log(value));

setTimeout(() => finish("done later"), 1000);

The outer promise is resolved to inner, but it does not fulfill until inner fulfills.

Gotcha 2: A promise cannot resolve to itself.
Self-resolution would create an impossible loop, so JavaScript rejects with a TypeError.
let resolveOuter;

const p = new Promise(resolve => {
  resolveOuter = resolve;
});

resolveOuter(p);

p.catch(error => {
  console.log(error instanceof TypeError);
});

// true
Gotcha 3: Thenables can be dangerous if badly written.
A thenable controls when it calls resolve or reject. JavaScript protects against multiple calls, but confusing thenables still make code harder to reason about.
const chaoticThenable = {
  then(resolve, reject) {
    resolve("first");
    reject(new Error("second"));
    resolve("third");
  }
};

Promise.resolve(chaoticThenable)
  .then(value => console.log(value));

// first

Only the first settlement wins. The later calls are ignored. JavaScript has guardrails, because apparently it has met developers.

Gotcha 4: Returning inside .then() matters.
If you create a promise inside a handler but forget to return it, the chain does not wait for it.

Broken

Promise.resolve()
  .then(() => {
    Promise.resolve("work done");
  })
  .then(value => console.log(value));

// undefined

Correct

Promise.resolve()
  .then(() => {
    return Promise.resolve("work done");
  })
  .then(value => console.log(value));

// work done
Gotcha 5: Flattening is not array flattening.
Promise flattening follows async state. It does not flatten data structures inside fulfillment values.
Promise.resolve([[1], [2], [3]])
  .then(value => console.log(value));

// [[1], [2], [3]]

5. Best Practices

Return promises from chains

When a .then() starts async work, return that promise so the chain assimilates it and waits for its result.

Prefer async/await for long flows

Promise mechanics still apply, but the code reads more like direct control flow. Fewer nested callbacks, fewer tiny traps wearing fake mustaches.

Be cautious with custom thenables

Use real promises unless you have a specific interop reason. A random object with then can trigger assimilation unexpectedly.

Use throw or rejection deliberately

Throwing inside a handler rejects the next promise. Returning a value from catch recovers the chain.

Clean chain pattern

getUser(userId)
  .then(user => getOrders(user.id))
  .then(orders => summarizeOrders(orders))
  .catch(error => {
    console.error("Workflow failed:", error);
    throw error;
  });

Each handler returns the next async operation. The chain remains flat and readable because promise resolution assimilates each returned promise.

6. Summary

Final rule: Promise resolution does not merely store a value. It evaluates the value. Plain values fulfill. Promises and thenables are assimilated. The visible result is flattening.
