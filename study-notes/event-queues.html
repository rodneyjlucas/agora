🧠 The Mental Model

Three Execution Contexts

1. Call Stack - Synchronous Code
Your regular, synchronous JavaScript code. Executes immediately, line by line. When empty, event loop checks what's next.

2. Microtask Queue - Promises & Async Operations
Promise callbacks (.then, .catch, .finally), async/await, and MutationObserver callbacks. ALL microtasks drain before ANY macrotask executes.

3. Macrotask Queue - Timers, Events, I/O
setTimeout, setInterval, events (click, scroll), file I/O, WebSocket callbacks. ONE macrotask executes per event loop cycle.

The Loop:
Call Stack → (empty?) → Microtask Queue (drain ALL) → Rendering → Macrotask (execute ONE) → Repeat

Why This Matters

Most developers think "async = next". But JavaScript has two types of async:

  • Microtask async: Promises (high priority)
  • Macrotask async: Timers, events, I/O (lower priority)

A Promise will always execute before setTimeout, even if setTimeout was registered first!

⚠️ Non-Intuitive Areas & Gotchas

🎯 Gotcha #1: setTimeout(0) is NOT "run now"

Many developers think setTimeout with 0ms delay will run immediately. It doesn't.

setTimeout(() => console.log('A'), 0);
Promise.resolve().then(() => console.log('B'));

Output: B, then A (not A, then B!)

Why? setTimeout callbacks are MACROTASKS. Promise callbacks are MICROTASKS. ALL microtasks run before ANY macrotask.

🎯 Gotcha #2: WebSocket events are NOT the same as fetch

WebSocket callbacks (onopen, onmessage) are MACROTASKS. But fetch() returns a Promise, so its .then() callbacks are MICROTASKS.

fetch('/api').then(() => console.log('A'));
ws.onmessage = () => console.log('B');

Output: A, then B (fetch is faster because Promises are higher priority!)

🎯 Gotcha #3: Promises inside macrotask callbacks create new microtask phase

When you create a Promise inside a macrotask callback, its .then() doesn't execute immediately. It queues to the NEXT microtask phase.

setTimeout(() => {
console.log('A');
Promise.resolve().then(() => console.log('B'));
}, 0);

Output: A, then B (but B is a new microtask phase, so other microtasks from NEXT cycle run before B)

🎯 Gotcha #4: Node.js process.nextTick vs browser's behavior

process.nextTick in Node.js is NOT the same as setTimeout. nextTick has special priority—it runs almost immediately, before I/O callbacks.

Browser doesn't have nextTick. Different platforms, different rules!

🎯 Gotcha #5: Event handlers run as macrotasks, blocking Promises

If user clicks and triggers an event handler, that handler is a MACROTASK. If the user clicks during heavy Promise processing, the click will wait.

This is why long Promise chains can cause UI jank!

🎯 Gotcha #6: console.log timing can be misleading

Browser DevTools and console implementations can batch logs. The order you see might not match actual execution order. Always reason about the queues, not just what you see printed.

✅ Best Practices

1. Use async/await instead of .then() chains

async/await is syntactic sugar for Promises but it's clearer to read and reason about:

// Bad: Chained .then()
fetch('/api').then(r => r.json()).then(data => console.log(data));

// Better: async/await
async function getData() {
  const r = await fetch('/api');
  const data = await r.json();
  console.log(data);
}

2. Break up long Promise chains into smaller operations

Long chains of .then() keep the microtask queue busy, blocking user interactions and rendering. Keep microtask work small:

// Bad: Too much in microtask queue
Promise.all(bigArray).then(() => {
  // 10000 lines of processing...
});

// Better: Break into chunks
for (let chunk of chunks) {
  await processChunk(chunk); // Short microtask
}

3. Use requestAnimationFrame() for animations, not setTimeout()

requestAnimationFrame is a macrotask that runs right before rendering. Perfect for smooth animations:

// Bad: setTimeout doesn't sync with rendering
setInterval(() => update(), 16);

// Better: Synced with browser rendering
requestAnimationFrame(update);

4. Be careful with event listeners and Promise processing

Event listeners run as macrotasks. If your Promise processing takes too long, clicks/scrolls will lag:

// Bad: Long Promise processing blocks clicks
async function processAllData() {
  for (let item of millionItems) {
    await heavyProcessing(item); // One huge chain
  }
}

// Better: Yield to let events process
async function processAllData() {
  for (let i = 0; i < millionItems.length; i++) {
    await heavyProcessing(millionItems[i]);
    if (i % 100 === 0) await new Promise(r => setTimeout(r, 0)); // Yield
  }
}

5. Understand your platform's event loop

Browser and Node.js have different event loops. Write code that works on both or be explicit about platform-specific code.

6. Use Performance tools to measure

Don't guess. Use Chrome DevTools (Performance tab) to see what's actually blocking rendering. Look for long tasks in the macrotask queue.

📋 Summary

What You Now Understand

The Three Queues:

  • CALL STACK - Synchronous code. Executes immediately.
  • MICROTASK - Promises, async/await, MutationObserver. ALL drain before next macrotask.
  • MACROTASK - setTimeout, setInterval, events, I/O. ONE per cycle.

The Event Loop Order:

  1. Execute all CALL STACK code
  2. Drain ALL MICROTASK queue
  3. Render (if needed)
  4. Execute ONE MACROTASK
  5. Go back to step 1

Key Insights:

  • Promises always run before setTimeout, even if setTimeout was registered first
  • WebSocket and event callbacks are macrotasks (can block rendering)
  • Long microtask chains block user interactions
  • Browser and Node.js have different event loop phases
  • Understanding queues is essential for performance optimization
