3. Main Topics: Queues and Promise Behavior

Call stack

The script starts as one task. JavaScript executes the top-level code synchronously from top to bottom. No promise callback runs while this initial script is still executing.

Microtask queue

Promise reactions created by .then(), .catch(), and .finally() are scheduled as promise jobs. They run after the current synchronous script completes and before the browser moves to the next task.

Task queue

setTimeout(..., 0) does not mean “run immediately.” It means “schedule a timer task as soon as eligible.” It still waits until the current script and the microtask checkpoint are done.

Network / fetch

fetch() starts an asynchronous operation and returns a promise immediately. Its .then() callback cannot run until the fetch promise fulfills with a response.

4. How Each Excerpt Affects Queues and Output Order

4.1 Timer registration

setTimeout(() => {
  console.log('8: setTimeout callback');
}, 0);

This registers a timer with the host environment. The callback is not pushed onto the call stack immediately. When the timer becomes eligible, it becomes a task. It still waits behind the initial script and behind the microtask queue created by already-fulfilled promises.

4.2 First resolved promise

Promise.resolve('first')
.then(value => {
    console.log('2: First trivial promise', value);
});

Promise.resolve('first') creates an already-fulfilled promise. Calling .then() does not run the callback now. It queues a promise reaction microtask. Because this is the first trivial resolved promise registered, its callback is ahead of the later trivial promise callbacks.

4.3 Fetch chain

fetch('https://api.example.com/user/1')
.then(response => {
    console.log('4: Response received');
return response.json();
})
.then(user => {
    console.log('5: User data parsed', user.name);
return fetch('https://api.example.com/posts/' + user.id);
})
.then(response => response.json())
.then(posts => {
    console.log('6: Posts loaded', posts.length);
})
.finally(() => {
    console.log('6.5: Cleanup (finally)');
});

The first fetch() returns a pending promise. The chain is assembled immediately, but none of those callbacks run during the initial script. When the response arrives, the first .then() reaction becomes a microtask and logs 4. Returning response.json() pauses the chain until JSON parsing fulfills. Returning the second fetch() pauses it again until the posts response arrives.

Interview-level phrasing: .then() returns a new promise. If the callback returns a plain value, the next promise fulfills with that value. If the callback returns a promise, the next promise adopts that promise’s eventual state.

6. Non-intuitive Gotchas and Common Mistakes

Gotcha: setTimeout(..., 0) is not first

A zero delay does not beat already-queued promise microtasks. Timers are tasks; promise reactions are microtasks.

Gotcha: .then() does not run inline

Even when the promise is already fulfilled, the callback is deferred to a microtask.

Gotcha: fetch does not join the trivial promise group

The trivial promises are already fulfilled. The fetch promise is pending, so its continuation waits for network completion.

Gotcha: missing return breaks the chain

If you forget return response.json() or return fetch(...), the next .then() may run with undefined or before the intended async work is done.

Gotcha: .finally() does not receive chain data

A finally callback is for cleanup, not transformation. It does not receive posts as a normal argument.

Gotcha: no .catch()

If either fetch fails, JSON parsing fails, or the placeholder API is unreachable, this chain rejects. In production, add .catch() after .finally() or before it depending on desired cleanup/error behavior.

Common broken version: using braces in a .then() but forgetting return. Block-bodied arrow functions do not implicitly return. 
// Broken: the next then receives undefined.
.then(response => {
  response.json();
})
// Correct: return the promise so the chain waits.
.then(response => {
return response.json();
})
// Also correct: expression-bodied arrow implicitly returns.
.then(response => response.json())

7. Summary

The main lesson is that promise chaining is about ordering and adoption. The current script runs first. Already-fulfilled promise reactions run as microtasks after the script finishes. Timer callbacks run as tasks after the microtask queue drains. Fetch chains continue only when their pending promises settle.
