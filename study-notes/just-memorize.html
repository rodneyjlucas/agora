Just Memorize: JavaScript Quirks & Historical Bugs

A collection of JavaScript behaviors that don't make sense but exist for historical reasons. These quirks trip up every developer—memorize them to avoid surprises in production.

1. Type System Quirks

typeof null === "object" 
typeof null;        // "object" (NOT "null"!)
typeof undefined;   // "undefined"
typeof NaN;         // "number" (NaN is a number)
Why: This is a bug in JavaScript's original design. null was supposed to have type "null", but a bug in the original implementation returned "object" instead. It was never fixed for backward compatibility.
NaN !== NaN 
NaN === NaN;           // false
Object.is(NaN, NaN);   // true
Number.isNaN(NaN);     // true
isNaN("hello");        // true (coerces to number first)
Why: IEEE 754 floating-point standard defines NaN as not equal to itself. Use Number.isNaN() or Object.is() to properly check for NaN.
typeof function returns "function", but functions are objects 
typeof (() => {});     // "function"
typeof class {};       // "function"
typeof String;         // "function"

Array.isArray([]);     // true (because Array is also a function)
Why: Functions are objects in JavaScript, but typeof returns a special "function" type for callable objects. Classes and constructors are functions too.

2. Equality & Coercion Traps

[] == false is true (but [] === false is false) 
[] == false;        // true
[] === false;       // false

[] == "";           // true
[] == 0;            // true

"" == 0;            // true
"" == false;        // true
Why: The == operator performs type coercion. An empty array converts to an empty string, which converts to 0, which coerces to false. Always use ===.
null == undefined (but null === undefined is false) 
null == undefined;   // true
null === undefined;  // false

null == 0;           // false
undefined == 0;      // false
Why: null and undefined are treated as equivalent by the == operator specifically. This is the one case where == can be useful: value == null checks for both null and undefined.
[] + [] returns "", but {} + [] returns 0 
[] + [];             // ""
[] + {};             // "[object Object]"
{} + [];             // 0 (block statement + unary plus)
{} + {};             // NaN
Why: Context matters. {} at the start of a statement is parsed as a block, not an object literal. The + [] is unary plus on an array. Always use parentheses for clarity: ({} + []).

3. Floating Point Precision

0.1 + 0.2 !== 0.3 
0.1 + 0.2;                    // 0.30000000000000004
0.1 + 0.2 === 0.3;           // false

Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON;  // true
Why: JavaScript uses IEEE 754 64-bit floating-point representation. Decimal fractions like 0.1 cannot be represented exactly in binary. For financial calculations, use integers (cents) or a decimal library.
Very large integers lose precision 
9007199254740992 === 9007199254740993;  // true!
Number.MAX_SAFE_INTEGER;                 // 9007199254740991
Number.MIN_SAFE_INTEGER;                 // -9007199254740991

999999999999999999 === 1000000000000000000;  // true
Why: JavaScript numbers have 53 bits of precision. Beyond that, consecutive integers are indistinguishable. Use BigInt for large integers.

4. Variable & Function Hoisting

var declarations are hoisted, but not initialized 
console.log(x);  // undefined (not a ReferenceError!)
var x = 5;
console.log(x);  // 5

// This is what actually happens:
var x;           // hoisted, initialized to undefined
console.log(x);  // undefined
x = 5;
console.log(x);  // 5
Why: Hoisting is a result of JavaScript's two-pass compilation. Declarations are processed before execution. This is why code using var before declaration doesn't crash—use const/let instead.
Function declarations are fully hoisted 
console.log(foo());  // "hello" (works!)
function foo() {
  return "hello";
}

// But function expressions are not
console.log(bar);    // undefined
console.log(bar()); // TypeError: bar is not a function
var bar = function() { return "world"; };
Why: Function declarations are moved entirely to the top of their scope. Function expressions (assigned to variables) follow variable hoisting rules instead.
const/let have a Temporal Dead Zone 
console.log(x);  // ReferenceError!
let x = 5;

// let and const are NOT hoisted in the traditional sense
// They're in a "Temporal Dead Zone" until the declaration is reached
Why: let and const exist from the start of the block scope, but accessing them before the declaration line throws an error. This prevents the "undefined" footgun of var.

5. Array & Object Oddities

Array.length can be manually set 
const arr = [1, 2, 3];
arr.length = 2;
console.log(arr);    // [1, 2] (element 3 deleted!)

arr.length = 5;
console.log(arr);    // [1, 2, empty × 3]
console.log(arr[4]); // undefined
Why: length is not a special property—it's a regular property that controls the array size. Setting it to a smaller number deletes elements; setting it to a larger number creates empty slots (sparse array).
Sparse arrays skip undefined elements in iteration 
const sparse = [1, , 3];  // empty slot at index 1
sparse.forEach((val, i) => console.log(i, val));
// Logs: 0 1
//       2 3
// (skips the empty slot!)

for (let i = 0; i < sparse.length; i++) {
  console.log(i, sparse[i]);  // Logs all indices including undefined
}
Why: Array methods like forEach, map, and filter skip empty slots. Traditional for loops don't. This inconsistency causes bugs.
Object property order is mostly insertion order, except for integer keys 
const obj = { b: 2, a: 1, 1: "one" };
Object.keys(obj);  // ["1", "b", "a"] (integer keys first!)

const obj2 = { "1": "a", b: 2, "0": "c" };
Object.keys(obj2); // ["0", "1", "b"] (numeric order first, then insertion)
Why: Integer-like property keys (0-4294967295) are ordered by numeric value first, then non-integer keys follow insertion order. This is a legacy behavior.

6. this Binding Surprises

this in regular functions depends on how they're called 
function greet() {
  console.log(this);
}

const obj = { greet };
obj.greet();        // this = obj
greet();            // this = window (or undefined in strict mode)

const fn = obj.greet;
fn();               // this = window (lost the binding!)
Why: this is determined dynamically at call time, not definition time. Calling fn() instead of obj.greet() loses the context. Use arrow functions or .bind().
Arrow functions don't have their own this 
const obj = {
  name: "obj",
  arrow: () => console.log(this),      // this = parent scope
  method: function() { console.log(this); }  // this = obj
};

obj.arrow();   // this = window (or global)
obj.method();  // this = obj
Why: Arrow functions inherit this from their enclosing scope at definition time. They're useful for callbacks but wrong for object methods.
Constructor calls with new set this to the instance 
function User(name) {
  this.name = name;  // this = new empty object
}

const user = new User("Alice");
console.log(user.name);  // "Alice"

// Without new, this is window/undefined
User("Bob");  // this.name = "Bob" leaks to global scope!
Why: The new keyword creates an object and sets it as this. Forgetting new causes this to be the wrong value. Use classes or always check new.target.

7. String & Number Quirks

Strings have methods (autoboxing) 
const str = "hello";
str.toUpperCase();    // "HELLO" (works!)

// How? JavaScript automatically wraps it:
new String("hello").toUpperCase();

str.foo = "bar";
console.log(str.foo);  // undefined (the wrapper was discarded!)
Why: When you call a method on a primitive, JavaScript temporarily wraps it in an object (autoboxing), calls the method, then discards the wrapper. Assigning properties to primitives is useless.
Template literals can be tagged 
const name = "Alice";
const age = 25;

const message = `Name: ${name}, Age: ${age}`;
// Normal template literal

const tagged = tag`Name: ${name}, Age: ${age}`;
// The tag function receives the string parts and values separately
Why: Tagged template literals are a rarely-used feature where a function can intercept and process the template. Most developers never encounter them.
Number constructor behavior differs from Number function 
Number("42");           // 42
new Number("42");       // Number { 42 } (object, not primitive!)

Number.prototype.valueOf.call(new Number("42"));  // 42

new Number("42") == 42;   // true (coercion)
new Number("42") === 42;  // false (different types)
Why: new Number() creates a Number object, while Number() coerces to a primitive. Objects are rarely the right choice here—use primitives.

8. Scope & Global Issues

Missing var/const/let creates globals 
function accident() {
  x = 10;  // No var/const/let! Creates global x
}

accident();
console.log(window.x);  // 10 (oops!)
console.log(x);         // 10
Why: Without a declaration, assignment creates a property on the global object. This is almost never intentional. Use strict mode: "use strict"; to throw an error instead.
Arguments object in regular functions (not arrow functions) 
function sum() {
  console.log(arguments);  // Array-like object with all args
  // arguments is NOT an array, but it has length
}

sum(1, 2, 3);  // arguments = {0: 1, 1: 2, 2: 3, length: 3}

// Use rest parameters instead
function sum2(...args) {
  args.reduce((a, b) => a + b);  // args is a real array
}
Why: The arguments object exists for historical reasons. Modern code uses rest parameters instead. Arrow functions don't have arguments.
Closure variables outlive their function 
function makeCounter() {
  let count = 0;
  return function() {
    return ++count;
  };
}

const counter = makeCounter();
counter();  // 1
counter();  // 2
counter();  // 3
// count persists even though makeCounter returned!
Why: The inner function has a closure over count. As long as the function exists, the variable persists. This is powerful but can leak memory if you're not careful.
Function scope vs block scope (var vs let) 
function test() {
  for (var i = 0; i < 3; i++) {}
  console.log(i);  // 3 (var is function-scoped!)
}

function test2() {
  for (let j = 0; j < 3; j++) {}
  console.log(j);  // ReferenceError (let is block-scoped)
}
Why: var is function-scoped, not block-scoped. This causes variables to "leak" out of loops and conditions. Always use const/let instead.
The Pattern: Most JavaScript quirks exist because the language was designed in 10 days in 1995. Many features were added without breaking the existing behavior, so weird edge cases accumulated. Modern JavaScript (ES2015+) fixed many issues, but the old quirks remain for backward compatibility. Always use strict mode and modern syntax (const/let, arrow functions, classes) to avoid them.
